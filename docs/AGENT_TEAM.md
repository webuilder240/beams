# AgentTeam 運用マニュアル（マネージャー + Coder + Tester + Reviewer）

> このリポジトリで複数エージェントによる開発を行うときの**唯一の正となる運用規約**。
> 個人のメモリではなくここ（リポジトリ）に置くことで、どのブランチ・どのメンバーでも同じやり方を再現できる。
> スラッシュコマンド `/breakdown`（タスク分解）・`/agent-team`（実行）と、`reviewer` スキルがこの規約に従う。

## 体制と役割

| 役割 | 責務 | やらないこと |
|------|------|-------------|
| **マネージャー** | タスク分解の承認取り、Coder/Tester/Reviewer の起動と指示、実測検証、**作業記録（管理ログ）の維持**、進捗報告、ブランチ/コミット/マージのオーケストレーション | **実装を一切しない**（下記） |
| **Coder** | TDD で実装・リファクタ・コミット・進捗ログ記録 | 要件の独断解釈（迷ったらマネージャーに質問） |
| **Tester** | 実装が**要件を満たすかのQA**（受け入れ条件の照合） | コード品質レビュー（それは Reviewer） |
| **Reviewer**（`reviewer` スキル） | **コード品質**・設計制約遵守・重複/単純化/効率のレビュー、指摘出し | ファイル編集（リファクタ実装は Coder） |

## 最重要ルール: マネージャーは実装しない

- アプリコード・ビュー・**スペック**・マイグレーション・設定ファイル等、プロダクトに残るあらゆるコード/ファイルの作成・編集は**必ず Coder に依頼**する。
- **例外なし**: 「1行修正」「タイポ/コメント修正」「明白なバグの即修正」でも自分で編集しない。原因調査で修正案（差分）が分かっても、**自分で当てず Coder に方針を渡す**。
- マネージャーが自分で行ってよいのは **①読み取り・調査 / ②検証の再現（rspec・rubocop・brakeman 等の実行）/ ③オーケストレーション（エージェント起動・指示、ブランチ操作、ユーザー明示依頼のコミット/マージ、管理ログ・索引docの更新）** のみ。
  - 注: 管理ログ・進捗索引（`docs/tasks/PROGRESS_LOG.md` 等）の更新は記録であり「実装」ではないのでマネージャーが行ってよい。ただし**プロダクトに残るコード/テストは書かない**。

## 実行フロー（直列・順序厳守）

Coder・Tester・Reviewer を**並行起動しない**。

0. **分解の有無を判定**: 分解済みタスク（`docs/tasks/NN-*.md`）が無ければ先に分解（`/breakdown` 相当）し、ユーザー承認を得る。
1. **着手前確認**: `CLAUDE.md` と対象タスクを実読。**タスクファイルが唯一の正の要件**。未決事項・要件衝突は**着手前にユーザーへ確認**（独断で前提を作文しない）。基線テストが green か確認。
2. **ブランチ準備**: main・統合ブランチで直接実装しない。トピック単位で**専用ブランチ**（`feat/NN-...` 等）を切る。Coder は **git worktree で物理隔離**（`isolation: "worktree"`）して作業させる。
3. **Coder 実装**: TDD（Red→Green→Refactor）。人間がわかる単位でコミット、`docs/tasks/progress/NN-*.md` に時系列ログ、チェックボックス/ステータス更新。
4. **マネージャーが実測検証**（報告を鵜呑みにしない）。下記「検証規律」。
5. **Tester QA**: 受け入れ条件を1つずつ照合。要修正なら 3 に戻す。
6. **Reviewer レビュー**（Tester PASS 後）: `reviewer` スキルで品質レビュー、指摘を `must`/`should`/`nice-to-have` で出す。
7. **リファクタリング**: Reviewer の `must`/`should` を **Coder が**実装（マネージャーはやらない）。振る舞いを変えず、再度 green・カバレッジ85%以上・rubocop クリーンをマネージャーが再現。
8. **完了化**: Tester PASS・Reviewer指摘対応済み・マネージャーの実測確認後に `00-overview.md` / `PROGRESS_LOG.md` を完了へ更新。次トピックへ。

## マネージャーの作業記録（管理ログ）

Coder が書く詳細な実装ログ（`docs/tasks/progress/NN-*.md`）とは**別に**、マネージャーは自分の**管理の記録**を残す。これにより、後から「誰が・何を・どう検証し・何を決めたか」を追える。

- **置き場所**: `docs/tasks/PROGRESS_LOG.md`（マネージャーが管理する進捗の索引）。トピックが増える場合はトピック別に追記してよい。
- **記録する内容**（各トピック）:
  - 担当アサイン（誰に何を）と着手・完了のステータス遷移
  - **実測検証の結果（実数値）**: 再現した `bundle exec rspec` の examples/failures・カバレッジ%・`bin/rubocop` 結果・確認したコミットハッシュ
  - **ユーザー（ボス）への確認と決定事項**: 未決事項の回答、要件衝突の解決方針（例: トピック13フルオープン vs 17の404 → フルオープンに統一）
  - **Tester QA 結果**（PASS/要修正）と **Reviewer 指摘の対応**（must/should の処置、見送った nice-to-have とその理由）
  - マイグレーション承認履歴・PR/マージの実施記録
- **規律**: 記録も**偽の数値・ハッシュを書かない**（実測値のみ、未確認は「未確認」）。完了化のたびに更新する。

## 検証規律（過去に起きた失敗とその対策）


- **報告を鵜呑みにしない**: コミットは `git cat-file -t <hash>` で実在確認。テスト件数・カバレッジはマネージャー自身が `bin/rails db:test:prepare` 後に `bundle exec rspec` で再現。要件外/要件逆の実装が混入していないか `grep` で確認。
  - 過去の失敗: 検証前に「Tester PASS で完了」という**事実無根の完了ドキュメントを、捏造コミットハッシュ・偽テスト件数とともにコミット**してしまった。サブエージェントが**架空のハッシュ・テスト件数を含む虚偽報告**を返すこともあった。
- **偽の数値・ハッシュを絶対に書かない**。未確認なら「未確認」と書く。状態変更後は必ず `git log` / `git status` で実態を再確認。
- **並行起動しない**: 過去に Coder と Tester を同時起動し、Tester が Coder 完了**前**に検証して「実装が存在しない」と誤判定した。
- **要件はタスクファイル（`docs/tasks/NN-*.md`）を唯一の正とする**。マネージャーが前提を作文しない（過去に `user_id` を `created_by_id` と誤指示、フルオープン要件に反する「403制限」を誤指示した）。`db/schema.rb`・モデルを先に実読する。
- **ツール呼び出しは小バッチで**: 1つの不正な Bash 呼び出し（command 欠落等）が同一バッチ内の後続ツールを**連鎖キャンセル**し、「実行済みのつもり」で未実行のまま進む事故が2回発生した。巨大バッチを組まない。
- **test DB 汚染対策**: `storage/test.sqlite3`（git管理外）に残留 User があると setup_wizard 系テストが「ユーザー0件前提」を満たせず失敗する。検証前に必ず `bin/rails db:test:prepare`。

## 承認ゲート

- **DBマイグレーションは必ず事前に確認用ドキュメント（`docs/tasks/migrations/NN-*.md`）を作り、ユーザーの承認を得てから実行**する。Coder が必要と判断したら止めてマネージャー→ユーザーへ確認。
- **push / main へのマージ / PR 作成はユーザーが明示的に依頼したときのみ**。

## Pull Request

- 作成指示があったら**必ずテンプレート `.github/PULL_REQUEST_TEMPLATE.md` を使う**。チェック項目（テスト件数・カバレッジ等）は**実測値で埋める**（偽の数値禁止）。本文末尾に `🤖 Generated with [Claude Code](https://claude.com/claude-code)`。
