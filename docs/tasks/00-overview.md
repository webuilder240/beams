## コーディング規約・制約条件（全トピック共通・必読）

すべてのトピック・全タスクにこの制約が適用される。各タスクの「受け入れ条件」はこの規約を満たすことを含む。

1. **TDD で実装する**（Red → Green → Refactor）。先に失敗するテスト（RSpec）を書き、それを通す最小実装を行い、その後リファクタする。**テストを後追いで書かない**。
2. **テストが通過するまでタスクを「完了」にしない**。`- [ ]` をチェック済みにする条件は「対象の RSpec が green、かつ既存テストを壊していない」こと。
3. **service クラス／`app/services` ディレクトリを作らない（禁止）**。`*Service` という命名も使わない。ドメインロジックの置き場所は次のいずれか:
   - 関連する **Active Record モデルのメソッド**（例: `Connection#bigquery`, `Query#bound_sql`）
   - **PORO（プレーンオブジェクト）を `app/models/` 配下に置く**（例: `app/models/dry_run.rb`、テストは `spec/models/`）
   - 運用スクリプト（バックアップ等）は **`lib/` 配下のモジュール＋rake/`bin` ラッパー**（テストは `spec/lib/`）
4. **カバレッジ 85% 以上を維持する**（SimpleCov）。`spec/spec_helper.rb` の閾値を **85%** に設定する（`CLAUDE.md` の制約と一致）。

## 推奨着手フェーズ

依存関係を踏まえた現実的な進め方。フェーズ内は概ね並行可能。

- **フェーズ0（土台）**: `01` — リネーム・暗号化・importmap準備・テスト基盤。**全ての前提**。
- **フェーズ1（アカウントと接続）**: `03` → `04` → `05`。ここまでで「ログインしてBigQueryに繋がる」状態。
- **フェーズ2（クエリを書く）**: `06` → `07`。スキーマを見ながらSQLを書いて保存できる。
- **フェーズ3（実行と差別化）**: `09` → `10`（実行の核）、並行して `08`（コスト保護＝目玉）。ここで「クエリを安全に走らせて結果が出る」。
- **フェーズ4（見せる・共有する）**: `11` → `12`、`13`・`14`（小トピック）。可視化・ダッシュボード・一覧。
- **フェーズ5（配布・運用）**: `02`（ONCE単一コンテナ）→ `15`（SQLite自動バックアップ・復旧）。`02` の `/storage`・worker・`recurring.yml` が前提なので最後にまとめて仕上げる。
- **フェーズ6（次フェーズ・利便性）**: `20` SSO（Google OAuth）→ `21` SQL本文LIKE検索 → `22` Redash SQLインポート。ユーザー指示（2026-06-05）でこの順。20はマイグレーション・設定方針の事前確認が必要。

> MVPの最短ゴール（デモ可能ライン）: `01→03→04→07→09→10→08`。スキーマブラウザ(06)・可視化(11)・ダッシュボード(12)は後追い可能。

## 全体の未決事項・要確認（横断）

実装前に方針決定が必要なもの。トピック個別の細かい未決は各ファイル末尾参照。
**A・B・C・E は2026-05-31に決定済み（下記）。D・F・G・H は各トピック着手時に確認。**

### A. JS資産の取り込み方針（importmap × Nodeなし）★複数トピックに影響 → ✅決定
- **決定（2026-05-31）: CDN pin で統一**。CodeMirror 6（07）・Chart.js（11）ともに `bin/importmap pin` でCDN（jspm等）にピン留めする。vendoringは行わない。
- 留意: 初回ロードに外部ネットワーク依存が残るため、ONCEのオフライン運用が要件化した場合は将来vendoringへ移行する余地を残す（→ 各トピックの未決事項に記録）。

### B. 結果の圧縮形式（10） → ✅決定
- **決定（2026-05-31）: JSON + gzip（`Zlib`）**。追加gem不要。MessagePackは採用しない。二重上限（10MB）内なのでバイト効率劣化の実害は小さい。

### C. CSV「全件」ダウンロードの実現方法（10, 11） → ✅決定
- **決定（2026-05-31）: 実行時に全件CSVも別途保持**。表示用の先頭N行blobとは別に、全件CSVを `/storage` 配下に保存し、ダウンロードは即時・再課金なし。容量増（=ONCEバックアップ増）は許容する。

### D. コスト換算レートの持ち方（08）
- GB単価をグローバル設定（`ApplicationSetting`）にするか `Connection` 単位にするか。

### E. パスワードリセット / ユーザー管理ルーティング（03）
- **決定（2026-05-31）: メール送信のパスワードリセットは作らない。adminがユーザーのパスワードを再発行（直接リセット）するUIで代替**。外部メール依存を避けONCE思想と整合。
- ユーザー管理を `/admin/users` 名前空間にするかフラットにするか（未決・03で確認）。

### F. SimpleCov カバレッジ閾値（01） → ✅決定
- **決定: 85% を維持**（`CLAUDE.md` のコーディング制約条件・本タスク群とも 85%。`spec/spec_helper.rb` を 85% に設定）。

### G. ONCE配布の細部（02）
- once-campfire `bin/boot` 流用時のライセンス表記。
- `db:prepare` を4DBへ流す正確なコマンド形式。
- 既存 `bin/docker-entrypoint`（Kamal用）を残すか廃止するか。

### H. テストにおけるBigQuery実APIの扱い（04, 05, 06, 08, 10）
